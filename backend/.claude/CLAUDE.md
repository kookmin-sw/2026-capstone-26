# 저장소 가이드라인

## 목적

이 파일은 `backend/`에서 작업하는 기여자와 코딩 에이전트를 위한 로컬 작업 규칙을 정의합니다.

## 핵심 규칙

- 작업이 명시적으로 동작 변경을 요구하지 않는 한 기존 동작을 보존합니다.
- 대규모 재작성보다 작고 점진적인 변경을 선호합니다.
- 단계적으로 작업을 진행한 후 다음에 무엇을 해야 하는지 알려주세요.
- 예외 메시지나 로그는 한국어로 작성합니다.

## Java 스타일

- 기본 기준선으로 Google Java Style을 따릅니다.
- 합리적으로 가능한 경우 메서드 매개변수를 한 줄에 유지합니다.
- 각 매개변수를 별도의 줄에 배치하는 방식으로 메서드 선언을 포맷하지 않습니다.
- 짧은 생성자 호출이나 예외 throw는 한 줄로 유지합니다.
- 정말 길 때만 줄바꿈합니다.
- 줄바꿈이 필요한 경우, 주변 파일의 스타일을 우선적으로 따릅니다.

금지:

```java
public void example(
    TypeA a,
    TypeB b,
    TypeC c
) {
}
```

권장:

```java
public void example(TypeA a, TypeB b, TypeC c) {
}
```

- 줄을 감싸야 하는 경우, 주변 파일에서 이미 사용된 스타일을 따릅니다.

## 작업 스타일

- 편집하기 전에 관련 코드와 컨텍스트를 조사합니다.
- 새로운 추상화를 도입하기 전에 기존 프로젝트 패턴을 재사용합니다.
- 데이터 조사를 위해 인터넷 검색 및 기타 리서치를 적극 활용합니다.
- 업무 태도에서 낙관주의보다 객관성을 우선시합니다.
- 작업할 때 부족한 지식과 개념이 있는지 검토합니다. 익숙하지 않거나 관련 지식과 경험이 없는 개념을 발견하면, 정의·기능·목적·예시·예상 효과 등 필요하다고 생각되는 모든 정보를 조사하고 작업에 종합적으로 활용합니다. 조사한 정보에서 또 다른 낯선 개념이 있으면, 그 개념에 대해서도 동일하게 합니다.
- 다양한 개선 사항 중 프로젝트를 더 안전하고 더 낫게 만드는 방법을 생각하고 개선 사항을 제시합니다.
- 테스트 메서드 이름은 한국어로 작성합니다.

## 금지 변경 사항

- 명시적인 요청 없는 대규모 구조적 변경
- 기존 어노테이션을 절대 삭제하지 않습니다.
- 명령이 없으면 테스트 코드를 건드리지 않습니다.

## 응답 기대사항

최종 응답에는 다음을 포함해야 합니다:

- 주요 작업 내용 요약
- 주요 구현 세부사항
- 남은 위험 요소 및 개선 사항
- 이후에 해야 할 일
